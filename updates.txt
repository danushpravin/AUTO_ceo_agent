1. Seasonality + demand trends (World Engine)
Do this first because everything else depends on data quality. Right now your world generates flat noisy demand — no Q4 spikes, no product decay, no launch curves. Before you add smarter AI reasoning on top, the underlying data needs to tell richer stories. If the world is flat, AUTO will always produce similar analyses regardless of what you ask. This is the foundation.
2. Scenario diff tool (Analytics + Tools layer)
Do this second because once your world has trends and seasonality, detecting structural breaks becomes genuinely meaningful. A scenario diff tool on flat data is answering a trivial question. On seasonal data with shocks injected, it becomes a real signal — "Google margin dropped 18 points in 7 days, this is not seasonal, this is structural." This also directly improves AUTO's output quality without touching the memory system.
3. Flag-driven RAG retrieval (RAG layer)
Do this third. Right now retrieval is driven by the user's query string. Once you have richer tool outputs from a more complex world, you want retrieval triggered by the actual flags returned — so if SPEND_SPIKE_WEAK_RETURN fires on Google post-seasonality-shift, the retriever pulls marketing efficiency knowledge automatically, not just when the user happens to ask about marketing. This is a targeted upgrade to agent.py and tools.py.
4. Session memory (Memory layer)
Do this last because it's the most architecturally complex and only becomes valuable when the underlying analysis is rich enough to be worth remembering. Memory on top of flat data and simple analysis is just storing redundant summaries. Memory on top of seasonal trends, shock detection, and flag-driven RAG is genuinely useful — AUTO can say "three weeks ago I flagged Influencers as loss-making, spend has since increased 40%, the recommendation was not acted on."
The general principle: fix the world first, then fix the analytics, then fix the retrieval, then add memory. Each layer depends on the one below it being solid.
